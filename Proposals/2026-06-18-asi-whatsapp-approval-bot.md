## שלב 1 — הבנת התרחיש (זרימת האישור מקצה לקצה)

במילים שלי, הלולאה שניר רוצה:

1. **נכס נוצר.** סוכן בצוות (יעל קופי / יובל ויזואל / מריו פוסט מטא) מסיים נכס שמיועד לפרסום.
2. **הנכס נכנס לתור.** נרשמת **רשומה חדשה ב-Airtable** בטבלת "תור תוכן" עם סטטוס `ממתין`, הקופי, קישור לויזואל, סוג הנכס, הסוכן שיצר, וערוץ היעד.
3. **אסי שולח לאישור.** אסי שולח לניר **הודעת וואטסאפ** עם הקופי + הויזואל + פרטי הפעולה, ושני כפתורים: **אשר** / **דחה**.
4. **ניר מחליט בוואטסאפ.** לחיצה אחת.
5. **אסי קולט את התשובה.** וובהוק נכנס ל-Make מזהה מי לחץ, על איזו רשומה, ומה בחר.
6. **עדכון + ביצוע:**
   - **אשר** → Airtable מתעדכן ל-`מאושר` → מופעל **תרחיש הפרסום הקיים** (id 6237903) → בסיום הסטטוס עובר ל-`פורסם` ואסי שולח אישור "פורסם ✓".
   - **דחה** → Airtable מתעדכן ל-`נדחה`, שום פרסום לא קורה.

**נקודת התורפה הטכנית של כל המהלך:** איך התשובה בוואטסאפ ("אשר" שניר לחץ) חוזרת **לרשומה הנכונה** ב-Airtable מבין עשרות רשומות בתור. זה נפתר ב-`whatsapp_message_id` + שדה מיפוי (פירוט בשלב 3).

## בחירת מסלול ה-WhatsApp (ההכרעה הקריטית)

### ההבחנה שחייבים להבין קודם

| | WhatsApp Business **App** (בטלפון) | WhatsApp Business **Platform / Cloud API** |
|---|---|---|
| מה זה | האפליקציה הירוקה הרגילה בטלפון | פלטפורמת API של Meta לשליחה אוטומטית |
| מתאים לאוטומציה? | **לא** — אין API רשמי, אסור לחבר בוטים | **כן** — זה מה שצריך לאסי |
| המספר | מספר הוואטסאפ הרגיל שלך | **מספר ייעודי שעובר ל-Cloud API** — לא ניתן להחזיק את אותו מספר בשתי המערכות |

⚠️ **חשוב מאוד לניר:** ברגע שמספר טלפון נרשם ל-Cloud API, **הוא נמחק מאפליקציית הוואטסאפ הרגילה**. לכן **לא** משתמשים במספר האישי/העסקי שאתה משתמש בו יומיום. צריך **מספר נפרד** (קו משני, eSIM, או מספר VoIP שעוד לא רשום בוואטסאפ).

### למה אסי הוא business-initiated — וזה קובע הכל

אסי **יוזם** את ההודעה לניר (לא ניר פותח שיחה). בוואטסאפ, הודעה שהעסק יוזם **חייבת להיות template מאושר** — אי אפשר לשלוח טקסט חופשי "קר". זה לא בירוקרטיה מיותרת; זה הכלל של Meta. החדשות הטובות (אומת 2026): template בקטגוריית **utility** (עדכון/בקשת פעולה, לא שיווק) נשלח **בחינם** כשהוא בתוך חלון 24 שעות פתוח, ובכל מקרה זול מאוד. הודעת אישור פנימית לניר היא utility מובהק.

### המסלולים

#### מסלול A — WhatsApp Business Cloud API (Meta הרשמי) דרך Make ⭐ **המלצה**

מודול Make מובנה "WhatsApp Business Cloud" עם הפעולות: **Send a Template** (להודעה היזומה עם כפתורים), **Send a Message** (טקסט חופשי בתוך חלון 24ש), ו-**Watch Events** (וובהוק לתשובות נכנסות).

- **יתרון:** ניר כבר עם Meta Business + Facebook + Instagram מחוברים ב-Make. אותו מסך ניהול, ללא ספק צד-שלישי. הזול ביותר לטווח ארוך. כפתורי quick-reply נתמכים דרך interactive template.
- **חיסרון:** הקמה ראשונית כבדה יותר — צריך App ב-Meta, מספר ייעודי + אימות, Phone Number ID, Permanent Token, ואישור template (24-48ש לרוב).
- **עלות:** הודעות utility — שבריר סנט עד חינם. בנפח של ניר (כמה אישורים ביום) — **זניח, פחות מ-1$ בחודש**.

#### מסלול B — Twilio WhatsApp (BSP מתווך)

Twilio מנהל את החיבור ל-Meta, ולמייק יש מודול Twilio.

- **יתרון:** הקמה מעט ידידותית יותר למתחילים, sandbox לבדיקות מהיר.
- **חיסרון:** **שכבת עלות נוספת** מעל Meta (~0.005$ להודעה + דמי Meta), עוד חשבון וחיוב לנהל, ועדיין דורש את אותו אימות מספר ו-template. כפילות עלות בלי יתרון אמיתי לניר שכבר בתוך אקוסיסטם Meta.

#### מסלול C — ביניים מהיר: שליחה חד-כיוונית + אישור דרך לינק (MVP זמני)

אסי שולח הודעה (וואטסאפ/מייל/טלגרם) עם **שני לינקים**: "✅ אשר" / "❌ דחה". כל לינק הוא וובהוק Make ייחודי שכבר נושא בתוכו את `record_id`. לחיצה = פתיחת דף → מפעיל את התרחיש.

- **יתרון:** **הכי מהיר להפעלה** — לא צריך template מאושר, לא צריך מספר ייעודי, אפשר אפילו להתחיל במייל/טלגרם תוך שעה.
- **חיסרון:** לא "כפתור בתוך וואטסאפ" אמיתי; דורש לחיצה שפותחת דפדפן. הלינקים חייבים להיות לא-ניתנים-לניחוש (token) כדי שלא יאושר בטעות.

### ההמלצה החד-משמעית

**להתחיל ב-C כ-MVP (השבוע), לעבור ל-A כיעד.**
הסיבה: C נותן לניר לראות את אסי **עובד מקצה לקצה תוך יום** — בלי להיתקע שבוע על אימות מספר ואישור template של Meta. במקביל ניר מתחיל את תהליך האימות של A ברקע. כשמסלול A מוכן, מחליפים רק את מודול השליחה/הקבלה — שאר הארכיטקטורה (Airtable, התרחיש הקיים) **זהה לחלוטין**. שום עבודה לא הולכת לפח.

> **על כפתורים אינטראקטיביים אמיתיים (A):** התשובה לכפתור quick-reply נכנסת כ-Watch Events webhook ל-Make. ב-payload יש את הטקסט/ה-ID של הכפתור שנלחץ + מספר השולח + `context.id` של ההודעה המקורית. ה-`context.id` הזה הוא מה שממפה את התשובה חזרה לרשומה (ראה שלב 3).

## שלב 2 — ארכיטקטורה (שרשרת המודולים)

מבנה של **שני תרחישים נפרדים** (best practice — שולח ומקבל לא מעורבבים, קל לתחזק ולכבות בנפרד):

### תרחיש A — "אסי שולח לאישור"

```
[1] Trigger: Airtable › Watch Records   (סטטוס = "ממתין")
        │
[2] Airtable › Get a Record  →  שולף קופי, ויזואל URL, סוג נכס, ערוץ יעד
        │
[3] Router (אופציונלי) → סוג ויזואל: תמונה / וידאו / טקסט-בלבד
        │
[4a] WhatsApp › Send a Template  (מסלול A)
     — template "asi_approval" עם משתנים: {{קופי}} {{סוג}} {{ערוץ}}
     — שני כפתורי quick-reply: "אשר" / "דחה"
     — (Header = image מה-URL של הויזואל)
   או
[4c] WhatsApp/Email › Send Message + שני לינקים עם token  (מסלול C)
        │
[5] Airtable › Update Record  →  שמירת whatsapp_message_id + סטטוס "נשלח לאישור"
```

### תרחיש B — "אסי מקבל תשובה"

```
[1] Trigger: WhatsApp › Watch Events  (וובהוק נכנס)   — מסלול A
   או   Custom Webhook  (לינק ה-token נלחץ)            — מסלול C
        │
[2] Filter: זו תגובת כפתור? (button reply / לא טקסט חופשי)
        │
[3] Airtable › Search Records  →  לפי whatsapp_message_id == context.id  (המיפוי הקריטי)
        │
[4] Router:
    ├─ "אשר" → Airtable Update (סטטוס "מאושר")
    │           → Make › Run a scenario (מפעיל את 6237903 "סוכן אורגני")
    │           → Airtable Update (סטטוס "פורסם") → WhatsApp Send "פורסם ✓"
    └─ "דחה" → Airtable Update (סטטוס "נדחה") → WhatsApp Send "נדחה, לא פורסם"
```

**למה שני תרחישים ולא אחד:** השליחה היא scheduled/instant על שינוי ב-Airtable; הקבלה היא webhook שמחכה לתשובה שיכולה להגיע דקות/שעות אחר כך. אי אפשר "להשהות" תרחיש Make שעות. הפרדה = כל אחד עם הטריגר הטבעי שלו.

## שלב 3 — מבנה ה-Airtable (טבלת "תור תוכן" / Content Queue)

הטבלה החדשה ב-Base הקיים (Airtable OAuth כבר מחובר, User usrPIKLbx7OrgufM0).

| שם שדה | סוג | תפקיד |
|---|---|---|
| `Asset Name` | Single line text | שם/כותרת הנכס לזיהוי מהיר |
| `Copy` | Long text | הקופי שיעל/מריו כתבו |
| `Visual` | Attachment **או** URL | התמונה/וידאו (ראה הערה למטה) |
| `Asset Type` | Single select | תמונה / וידאו / פוסט-מטא / מייל |
| `Source Agent` | Single select | יעל / יובל / מריו / אנצ'לוטי |
| `Target Channel` | Single select | Facebook / Instagram / שניהם |
| `Status` | **Single select** | `ממתין` / `נשלח לאישור` / `מאושר` / `נדחה` / `פורסם` |
| `whatsapp_message_id` | Single line text | **מפתח המיפוי** — ה-ID שאסי קיבל מ-WhatsApp בשליחה |
| `approval_token` | Single line text | טוקן ייחודי (מסלול C, ללינקים) |
| `Created At` | Created time | חותמת זמן אוטומטית |
| `Decided At` | Date/time | מתי ניר החליט |
| `Publish Result` | Long text | מה החזיר תרחיש הפרסום (כולל שגיאות) |

### המיפוי הקריטי — איך התשובה חוזרת לרשומה הנכונה

זו **הנקודה הטכנית החשובה ביותר בכל הפרויקט**. הזרימה:

1. בתרחיש A, כשאסי שולח, מודול ה-WhatsApp **מחזיר `message_id`** של ההודעה שיצאה.
2. אנחנו **שומרים אותו** בשדה `whatsapp_message_id` באותה רשומה.
3. כשניר לוחץ "אשר", הוואטסאפ שולח תשובה שכוללת `context.id` — **שווה ל-`message_id` של ההודעה המקורית**.
4. בתרחיש B, מודול `Airtable Search` מחפש את הרשומה ש-`whatsapp_message_id == context.id`. **מצאנו את הרשומה הנכונה — תמיד, בלי בלבול.**

> במסלול C זה פשוט עוד יותר: ה-`record_id` (או `approval_token`) **מוטמע בתוך כתובת הלינק עצמה**, אז כשניר לוחץ, הוובהוק כבר יודע על איזו רשומה מדובר — אין צורך בחיפוש.

## שלב 5 — בדיקה ומסירה

### איך בודקים שאסי עובד (לפני הפעלה חיה)

1. **Run once על תרחיש A** עם רשומת דמה ב-Airtable (`Status=ממתין`). לוודא שהודעה הגיעה לוואטסאפ עם הכפתורים והויזואל.
2. ללחוץ "אשר" בטלפון → לבדוק ב-execution log של תרחיש B שה-Search מצא את הרשומה הנכונה.
3. לוודא שהסטטוס ב-Airtable השתנה ל-`מאושר` → `פורסם`.
4. לבדוק את **ה-bundle** בלוג: שדה `context.id` קיים ושווה ל-`whatsapp_message_id`.
5. לחזור על אותו תהליך עם "דחה" → לוודא ש**לא** פורסם כלום.
6. בדיקת קצה: ללחוץ "אשר" פעמיים → לוודא שהפילטר חוסם את השנייה.

### מה ניר חייב לחבר בעצמו (דורש אימות זהות/טלפון — אי אפשר לעשות בשבילו)

**מסלול A (Cloud API מלא):**
- [ ] **מספר טלפון ייעודי** ל-WhatsApp — לא המספר היומיומי. (eSIM/קו משני/VoIP שלא רשום בוואטסאפ.)
- [ ] ב-Meta Business: להוסיף את המוצר **WhatsApp** ל-App, ולאמת את המספר (קוד SMS/שיחה).
- [ ] להעתיק את **Phone Number ID** ו-**WhatsApp Business Account ID** (Meta › WhatsApp › Getting Started).
- [ ] ליצור **Permanent Access Token** (System User token — לא token זמני של 24ש).
- [ ] להגיש את ה-**template** `asi_approval` (utility, עם 2 כפתורי quick-reply) לאישור Meta — לוקח 24-48ש.
- [ ] לאמת את מספר הטלפון של ניר כ**נמען** (שיקבל את ההתראות).

**מסלול C (MVP מהיר):** כמעט כלום — אם מתחילים במייל/טלגרם, אין אימות מספר בכלל.

### מה ראובן יכול לבנות לבד דרך ה-MCP (אחרי "אשר")

- [x] יצירת טבלת "תור תוכן" ב-Airtable עם כל השדות (Airtable MCP).
- [x] בניית תרחיש A ותרחיש B ב-Make (Make MCP), כולל מיפוי שדות, Router, Filter, Error Handler.
- [x] חיבור תרחיש B לתרחיש הפרסום הקיים (6237903).
- [x] במסלול C: יצירת הוובהוקים והטוקנים.

## הצעת שלב ראשון מינימלי (MVP)

**הכי הגיוני להפעיל ראשון:**

1. ראובן בונה את **טבלת "תור תוכן"** ב-Airtable (Airtable MCP).
