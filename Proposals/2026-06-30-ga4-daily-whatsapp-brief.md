**מי:** אמציה (אוטומציות) | **תאריך:** 2026-06-30 | **סטטוס:** תכנון בלבד — לא נבנה/הופעל חי. ממתין ל"אשר".
**מצב:** ליווי ולמידה — מוסבר צעד-צעד כדי שניר יוכל לשחזר לבד.

## 1. התרחיש כפי שהבנתי

כל ערב ב-20:00 (שעון ישראל), Make מתעורר לבד, מושך מ-Google Analytics 4 את נתוני **אתמול** (יום שלם, 00:00–23:59), מנסח אותם ל**בריף קצר וקריא בעברית**, ושולח אותו אליך בערוץ הודעות (WhatsApp / Telegram / Email). בלי שתצטרך להיכנס ל-GA4 בכלל — הדוח מגיע אליך.

**הקלט (Trigger):** שעון — כל יום ב-20:00.
**העיבוד:** קריאה ל-GA4 Data API → סידור המספרים → ניסוח טקסט.
**הפלט:** הודעת בריף אחת בערוץ שתבחר.

## 2. הארכיטקטורה (שרשרת המודולים ב-Make)

```
[1] Schedule          כל יום 20:00, אזור זמן Asia/Jerusalem
        │
        ▼
[2] GA4 · Run a Report (קריאה 1 — סיכום כללי)
        │   משתמשים, sessions, page views, engagement, scroll
        ▼
[3] GA4 · Run a Report (קריאה 2 — פילוח מקורות)   ← אופציונלי כקריאה נפרדת
        │   source/medium/campaign
        ▼
[4] GA4 · Run a Report (קריאה 3 — דפים מובילים)   ← אופציונלי כקריאה נפרדת
        │   pagePath / pageTitle
        ▼
[5] Tools · Set/Compose  — ניסוח הטקסט בעברית (מרכיב את הבריף ממוצא)
        │
        ▼
[6] ערוץ שליחה: WhatsApp / Telegram / Email
```

**למה שלוש קריאות נפרדות ולא אחת?** ב-GA4 Data API יש כלל ברזל: **כל שילוב dimensions+metrics חייב להיות ברמת scope תואמת.** אם תערבב dimension של דף (`pagePath`) עם metric של משתמשים (`totalUsers`) באותה קריאה — תקבל מספרים מטעים (כי GA4 "מכפיל" שורות). לכן מפצלים: כל "סעיף" בבריף = קריאה ממוקדת. זה גם **כלל הברזל מס' 1 שלנו** — תהליכים קטנים וממוקדים, קל לתחזק ולדבג.
*(אפשר גם קריאה אחת עם הרבה dimensions — אבל אז העיבוד מסובך יותר. לבריף יומי, 3 קריאות נקיות עדיפות למתחיל.)*

> 💡 **עיקרון ללמידה:** ב-GA4, **metric = מספר** (כמה), **dimension = חיתוך** (לפי מה). תמיד מתאימים את הרמה: dimensions של session עם metrics של session, dimensions של דף/אירוע עם metrics של אירוע.

## 3. בנייה צעד-אחר-צעד — ה-metrics וה-dimensions המדויקים

> כל השמות למטה **אומתו מול תיעוד GA4 Data API (v1beta)** — לא מהזיכרון. אלה ה-`apiName` המדויקים שהמודול דורש.

### מודול [1] — Schedule
- Run scenario: **Every day**, Time: **20:00**.
- ⚙️ באזור הארגון ב-Make → ודא **Time zone = Asia/Jerusalem**, אחרת 20:00 ירוץ ב-UTC (= 23:00 אצלך).

### מודול [2] — GA4 · Run a Report (קריאה 1: סיכום כללי)
- **Property:** ה-Property ID **המספרי** (ראה סעיף 5 — אתה צריך לספק אותו).
- **Date Range:** Start date `yesterday`, End date `yesterday`. *(GA4 מקבל את המילה `yesterday` כקיצור — אין צורך לחשב תאריך.)*
- **Metrics:**

| מה שרוצים | apiName מדויק |
|---|---|
| סה"כ משתמשים | `totalUsers` |
| משתמשים חדשים | `newUsers` |
| Sessions | `sessions` |
| צפיות עמוד | `screenPageViews` |
| Engagement rate | `engagementRate` |
| משך מעורבות ממוצע ל-session | `averageSessionDuration` |

- **Dimensions:** `newVsReturning` (כדי לראות פילוח חדש מול חוזר באותה קריאה).
- **Limit:** 10 שורות (מספיק — יהיו 2 שורות: new / returning).

### מודול [3] — GA4 · Run a Report (קריאה 2: מאיפה הגיעו)
- **Date Range:** `yesterday` → `yesterday`.
- **Metrics:** `sessions`, `totalUsers`.
- **Dimensions:** `sessionSourceMedium`, `sessionCampaignName`.
- **Order by:** `sessions` יורד. **Limit:** 8.

> ⚠️ **טעות נפוצה — קריטי:** ל-source/medium **חייבים** את הגרסה עם תחילית `session` (`sessionSourceMedium`, `sessionCampaignName`). ה-dimensions הגולמיים (`source`, `medium`, `campaignName`) **משויכים רק לאירועי conversion** ויחזירו 0 / נתון חסר עבור sessions רגילים. זו בדיוק הסיבה שה-UTM שלך (`instagram/social/bio-link`, `facebook/social/bio-link`) יופיע נכון רק עם `sessionSourceMedium`. *(אומת מול התיעוד.)*

### מודול [4] — GA4 · Run a Report (קריאה 3: דפים מובילים)
- **Date Range:** `yesterday` → `yesterday`.
- **Metrics:** `screenPageViews`.
- **Dimensions:** `pageTitle` (קריא יותר מ-`pagePath`; אפשר להוסיף גם `pagePath`).
- **Order by:** `screenPageViews` יורד. **Limit:** 5.

### מודול [5] — אינדיקציית גלילה (scroll)
שתי דרכים — בחר אחת:
- **דרך A (קריאה רביעית קטנה):** Run a Report — Metric `eventCount`, Dimension `eventName`, **Filter: `eventName` = `scroll`**, Date `yesterday`. מחזיר כמה אירועי scroll (= כמה פעמים מישהו גלל ל-90%).
- **דרך B (לחסוך Operation):** מאחדים את קריאה 5 לתוך קריאה כללית של אירועים — Metric `eventCount`, Dimension `eventName`, Limit 15. מקבלים שורה אחת `scroll` מתוך כל האירועים.

> 💡 **למה scroll נדרש קריאה נפרדת?** כי `eventCount` הוא metric ברמת **אירוע**, ו-`eventName` הוא dimension ברמת אירוע — אסור לערבב אותם עם `totalUsers` מקריאה 1 (סקופ שונה). זה שוב אותו כלל סקופ.

### מודול [6] — Tools (ניסוח הבריף בעברית)
מודול **Set multiple variables** או **Compose a string**. מרכיב טקסט ידידותי. דוגמת תבנית (במיפוי שדות אמיתי מהמודולים שמעליו):

```
📊 בריף NIRO — {{אתמול בעברית}}

👥 משתמשים: {{totalUsers}} (חדשים: {{newUsers}})
🔁 Sessions: {{sessions}} · צפיות עמוד: {{screenPageViews}}
⏱️ מעורבות: {{engagementRate}}% · זמן ממוצע: {{averageSessionDuration}}ש'

📍 מאיפה הגיעו:
{{לולאה על קריאה 2: source/medium — sessions}}

📄 דפים מובילים:
{{לולאה על קריאה 3: pageTitle — views}}

📜 גלילה ל-90%: {{scroll eventCount}} פעמים

לילה טוב 🌙
```

*(הניסוח הסופי הזה — אם תרצה שיהיה ממש מלוטש ובקול המותג — זה תפקיד של יעל. אני מסמן את זה לראובן; כרגע התבנית פונקציונלית.)*

## 4. נקודת ההחלטה המרכזית — ערוץ השליחה (השוואה כנה)

זו ההחלטה היחידה שחוסמת אותך מלהתניע. הנה 3 המסלולים בכנות מלאה:

| מסלול | מה צריך להקים | עלות | זמן הקמה | מתי מתאים |
|---|---|---|---|---|
| **WhatsApp** (מה שביקשת) | WhatsApp Business API דרך ספק (360dialog / Twilio / Meta Cloud API). דורש: מספר ייעודי, אימות עסק ב-Meta, ו**אישור Message Template** (הודעה יזומה למספר שלך = "utility template" שמטא צריכה לאשר, 1–3 ימי עבודה). | ~‏$0–5 לחודש בסיס + פרוטא-הודעה (אגורות); השליחה לעצמך זניחה | **גבוה** — ימים (אימות + אישור תבנית) | כשרוצים שזה יגיע לוואטסאפ דווקא, ומוכנים לתהליך הקמה חד-פעמי |
| **Telegram bot** | יצירת בוט דרך @BotFather (2 דקות), העתקת token, מודול Telegram ב-Make. שולח לעצמך/לקבוצה. | **חינם לגמרי** | **נמוך** — 10 דקות | כשרוצים בריף יומי לטלפון *היום*, בלי בירוקרטיה |
| **Email** | מודול Gmail/Email ב-Make (כבר מחובר אצלך — אסי משתמש בו). | **חינם** | **מיידי** — 0 הקמה | fallback מושלם, ותואם את תשתית אסי הקיימת |

### 🎯 ההמלצה שלי
- **להתנעה היום:** התחל ב-**Telegram** או **Email**. הלוגיקה של GA4 (מודולים 1–5) **זהה לכל הערוצים** — רק המודול האחרון משתנה. ככה תראה את הבריף עובד מחר בבוקר, בלי להמתין לאישורי Meta.
- **WhatsApp כשלב 2:** ברגע שהבריף עובד ואתה מרוצה מהתוכן, נקים את WhatsApp Business API במקביל ונחליף רק את מודול [6]. ה-90% של העבודה כבר יהיו מוכנים.

> זה בדיוק **כלל הברזל מס' 4** שלנו: Webhooks/מודולים מחברים כמעט הכל — ולכן החלפת ערוץ היא החלפת מודול אחד, לא בנייה מחדש.

**👉 ההחלטה שאני צריך ממך:** באיזה ערוץ מתניעים — **Telegram (מומלץ, מהיר וחינמי)**, **Email (הכי פשוט)**, או **WhatsApp עכשיו (עם זמן הקמה)?**

## 5. מה אתה צריך לספק לפני הבנייה

1. **Property ID מספרי** — GA4 → Admin (גלגל שיניים) → Property Settings → מספר כמו `123456789`. ⚠️ **לא** ה-Measurement ID `G-P22PS0TVHS` — הוא לא עובד ב-Data API. (זה הקלט שמייד דורש המודול.)
2. **הרשאת Google למשיכה** — חיבור OAuth של חשבון Google שיש לו גישת Viewer ל-property הזה ב-GA4. נעשה דרך מסך החיבורים של Make (OAuth 2.0).
3. **בחירת ערוץ שליחה** (סעיף 4) — ואם Telegram: token מ-@BotFather; אם WhatsApp: נתחיל תהליך הקמת ספק.

| בעיה אפשרית | סימן מזהה | פתרון |

| הזמן רץ ב-UTC במקום ישראל | הבריף מגיע ב-23:00 | הגדר Time zone של הארגון/התרחיש ל-`Asia/Jerusalem` |
| מספרי source/medium יוצאים 0 | טבלת מקורות ריקה | להשתמש ב-`sessionSourceMedium` (לא `source`) — תוקן בתכנון |
| ערבוב סקופים מחזיר מספרים מנופחים | סה"כ משתמשים גבוה מהמציאות | קריאות נפרדות לכל סקופ (כבר בארכיטקטורה) |
| GA4 API לא מגיב / 429 rate limit | Scenario נכשל | Error Handler עם **Retry** (3 ניסיונות, השהיה 30ש') על מודולי GA4 |
| יום בלי תנועה → שדה ריק שובר טקסט | בריף עם `[empty]` | Filter/ברירת מחדל "0" בניסוח (מודול Tools) |
| GA4 "data freshness" — נתוני אתמול עדיין מתעדכנים | מספרים זזים קצת | 20:00 על *אתמול* בטוח (נתוני יום קודם כבר סופיים) — תקין |
| חיבור OAuth ב-Make במצב Testing | דורש חידוש שבועי | להעלות את ה-OAuth consent ל-Production ב-Google Cloud |

## 7. עלות ובדיקה

- **Operations לכל הרצה:** ~4–6 (Schedule 0 + 3–4 קריאות GA4 + Tools 1 + שליחה 1). פעם ביום = **~120–180 Operations בחודש**. זניח לחלוטין בכל תוכנית Make (גם החינמית).
- **בדיקה לפני הפעלה:** **Run once** ידני → לבדוק שה-3 קריאות מחזירות נתונים → לבדוק שהטקסט מתנסח נכון → לבדוק שההודעה מגיעה לערוץ. רק אז להפעיל את ה-Schedule.
