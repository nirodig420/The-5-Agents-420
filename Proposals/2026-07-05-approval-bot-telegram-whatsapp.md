## שלב 2 — ארכיטקטורה

### הכרעה 1: **Telegram** (לא Green API) — המלצה חדה

| שיקול | Telegram Bot (נבחר ⭐) | Green API / וואטסאפ (נדחה לע"ע) |
|---|---|---|
| **כפתורי אישור** | inline keyboard רשמי ויציב; ה-`callback_data` נושא את ה-record id → **מיפוי ודאי לשורה הנכונה** | Green API מציעים `SendInteractiveButtons`, אבל **בתיעוד שלהם עצמם**: היציבות "לא תלויה ב-Green API", ההודעה "עלולה לא להופיע" בחלק מהגרסאות, וההמלצה שלהם — "תמיד לשכפל כפתורים בהודעה רגילה". לשער שמדליק פרסום אמיתי — לא מספיק אמין |
| **מיפוי תשובה→שורה** | דטרמיניסטי: הכפתור עצמו מכיל `approve\|recXXX` | תשובת טקסט חופשי "אשר" — **דו-משמעית** כששני פוסטים ממתינים; מיפוי דרך reply-מצוטט (`stanzaId`) שביר ותלוי בזה שניר יזכור לצטט |
| **עלות** | חינם לחלוטין, ללא מגבלת הודעות | מנוי אינסטנס חודשי מתמשך |
| **מה כבר קיים אצלנו** | בוט `niroGA4_daily_brief_bot` + חיבור Telegram ב-Make + chat id של ניר (`352155152`) — **מתרחיש 6384652 שעובד** | חיבור Green API קיים (תרחיש 5618188) — אבל רק לשליחה יוצאת |
| **אבטחה** | callback מגיע רק דרך הבוט; מסננים לפי user id של ניר | וובהוק נכנס פתוח יותר, אימות ידני |

**השורה התחתונה:** ללולאת אישור שמדליקה פרסום אמיתי צריך כפתור **אמין ב-100%** עם מיפוי ודאי — זה Telegram. ניר כבר מקבל שם את בריף ה-GA4, אז זה לא ערוץ זר. **Green API נשאר בתמונה** כאופציית שלב 2: הודעת "שיקוף" חד-כיוונית לוואטסאפ ("נשלח אליך אישור בטלגרם") — בלי לוגיקת החלטה.

### הכרעה 2: מבנה — שני תרחישים נפרדים

בדיוק כמו בארכיטקטורת אסי הנעולה (שולח ומקבל לא מתערבבים — לכל אחד הטריגר הטבעי שלו, ואפשר לכבות/לדבג כל אחד בנפרד):

```
תרחיש C — "אסי-טלגרם: שולח לאישור"
[Airtable Watch Records: סטטוס="נשלח לאישור" + טרם-נשלח]
   → Router: יש ויזואל?
      ├─ כן → [Telegram Send a Photo] → [Telegram Send Message + 2 כפתורים] → [Airtable Update: telegram_message_id]
      └─ לא → [Telegram התראה לניר: "שורה בלי ויזואל — לא נשלחה"]

תרחיש D — "אסי-טלגרם: קולט החלטה"   (מיידי — Webhook)
[Custom Webhook ← Telegram callback]
   → Filter: זה callback? זה ניר?
   → [Airtable Get a Record] → Filter: סטטוס עדיין "נשלח לאישור"?
   → Router לפי callback_data:
      ├─ approve → [Update: סטטוס="מאושר" + ✅checkbox + אושר בתאריך] → [Edit Message: "✅ אושר — יפורסם א'/ג'/ה' 08:45"]
      │              └→ השורה נכנסת אוטומטית ל-View "✅ מאושרים" + תרחיש הפרסום קולט
      └─ reject  → [Update סטטוס="נדחה"]  → [Edit Message: "❌ נדחה"]  → View "❌ נדחו"
```

**איך מזהים לאיזו שורה שייכת הלחיצה (הנקודה הקריטית):** בכל כפתור מוטמע
`callback_data` בפורמט `approve|recXXXXXXXXXXXXXX` (ההחלטה + ה-record id של Airtable).
כשניר לוחץ, טלגרם מחזיר את המחרוזת הזו בדיוק — התרחיש הקולט מפרק אותה (`split`) ויודע
בוודאות איזו שורה לעדכן. (מגבלת callback_data = 64 בייט; אצלנו ~25 — בטוח.)
**זו למידה חשובה:** בטלגרם ה"הקשר" נוסע *בתוך הכפתור* — אין צורך בטבלת מיפוי, בניגוד לוואטסאפ שם צריך לשמור message id ולהצליב.

### הכרעה 3: קליטת הלחיצה — **Custom Webhook + setWebhook**, לא "Watch Updates"

מודול Telegram "Watch Updates" של Make הוא **polling** (בדיקה מתוזמנת): גם תגובה איטית
(עד מרווח הבדיקה) וגם **עלות** — כל בדיקה = 1 op גם כשאין כלום (בדיקה כל 5 דק' ≈ 8,600
ops/חודש — לא בא בחשבון). בקהילת Make זו גם נקודת חיכוך ידועה עם callback queries.

**הפתרון המקצועי:** Custom Webhook רגיל ב-Make + פעולה חד-פעמית של רישום ה-webhook אצל
טלגרם (`setWebhook`) — פתיחת URL אחד בדפדפן, 30 שניות. מאותו רגע טלגרם **דוחף** כל לחיצה
ישירות לתרחיש: **מיידי, ו-0 ops כשאין לחיצות.**
(מגבלה מובנית של טלגרם: בוט עם webhook לא יכול במקביל להשתמש ב-polling — לא מפריע לנו,
כי תרחיש ה-GA4 רק *שולח* דרך הבוט הזה, לא מאזין.)

### הכרעה 4: בוט — **ממחזרים את `niroGA4_daily_brief_bot`** (אפס הקמה)

הבוט, הטוקן, החיבור ב-Make וה-chat id של ניר — הכל כבר קיים ועובד. יצירת בוט ייעודי
("ניקיון") אפשרית ב-2 דקות ב-BotFather, אבל מוסיפה חיבור + טוקן לנהל בלי רווח אמיתי.
אם בעתיד הבוט "יתמלא" — נפצל.

3. **אישור הוספת 3 השדות** לטבלה B (§3.0 — פעולת Airtable MCP, הוספתית והפיכה).

## שלב 5 — בדיקה ומסירה (לפני שום פוסט אמיתי)

1. **שורת דמה** בטבלה B: נושא "בדיקת אסי-טלגרם", תמונת בדיקה, קופי קצר, ערוץ FB, סטטוס "נשלח לאישור". **חשוב:** לוודא שתרחיש הפרסום 6237903 **כבוי (OFF)** בזמן הבדיקות — שהדמה לא תתפרסם באמת.
2. **Run once על תרחיש C** → בטלפון: תמונה + קופי + 2 כפתורים. בלוג: מודול 5 כתב `telegram_message_id`.
3. **Run once נוסף על C** → הדמה **לא** נשלחת שוב (הנוסחה מסננת). ✓ אנטי-כפילות שליחה.
4. תרחיש D **ON** (אחרי setWebhook) → לחיצה על **✅ אשר**: בלוג D — הפילטרים עברו, נתיב approve; ב-Airtable — סטטוס "מאושר" **+ ה-checkbox מסומן + "אושר בתאריך" מלא**, והשורה **מופיעה ב-View "✅ מאושרים"**; בטלגרם — ההודעה נערכה ל"אושר" והכפתורים נעלמו.
5. **לחיצה חוזרת** על ההודעה הישנה (אם עוד יש כפתורים בגרסה קודמת) → Filter D-5 חוסם. ✓
6. **בדיקת דחייה:** איפוס הדמה ("נשלח לאישור" + ריקון `telegram_message_id`) → Run C → **❌ דחה** → "נדחה" בטבלה, "נדחה" בטלגרם, שום פרסום. ✓
7. **בדיקת זר:** שליחת הודעת טקסט רגילה לבוט → בלוג D: Filter D-2 חסם. ✓
8. **מחיקת שורת הדמה**, הדלקת 6237903 חזרה (ON), הפעלת C ו-D לצמיתות — ומהפוסט הבא של הבאצ' השבועי, האישור כולו מהטלפון.

## סיכום — מה / איך / למה / סיכון

- **מה:** שני תרחישי Make חדשים ("אסי-טלגרם שולח" + "אסי-טלגרם קולט") + 3 שדות ו-3 Views ב-Airtable, שהופכים את האישור ללחיצת כפתור בטלגרם — וכל אישור נכנס אוטומטית ל"טבלת המאושרים" (View "✅ מאושרים", כמו "לקוחות משלמים" ב-CRM). תרחיש הפרסום שהוכח היום — ללא שינוי.
- **איך:** כפתורי inline עם record id בתוך ה-callback; webhook מיידי לקליטה; Update אחד = סטטוס "מאושר" (מדליק את הצינור הקיים בתזמון א'/ג'/ה' 08:45) + checkbox + תאריך אישור (מזינים את ה-View).
- **למה Telegram:** כפתורים אמינים + מיפוי ודאי + חינם + כל התשתית כבר קיימת. Green API נדחה כשער-החלטה בגלל אי-יציבות מוצהרת של כפתורים ומיפוי עמום של טקסט חופשי; יוכל לשמש כהודעת שיקוף בשלב 2.
- **סיכון/עלות:** ~1,700 ops/חודש (polling C בחלון שעות) + ~10 ops לפוסט — הרחק מהמכסה. אפס נגיעה בצינור חי. כל הפעולות הפיכות; ההפעלה החיה אחרונה ורק אחרי בדיקות.
